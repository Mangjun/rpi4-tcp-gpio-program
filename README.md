# TCP 기반 원격 장치 제어 시스템

## 1. 프로젝트 개요

#### 1.1 프로젝트 소개
본 프로젝트는 TCP/IP 소켓 통신을 기반으로 원격지에 있는 임베디드 장치(라즈베리파이 4)에 연결된 다양한 하드웨어 모듈을 제어하고 모니터링하는 시스템입니다.

다중 클라이언트 접속을 효율적으로 처리하기 위해 멀티스레드 기반으로 설계되었으며, 하드웨어 제어 로직을 독립적인 동적 라이브러리(.so) 형태로 구성하여 유지보수성과 확장성을 극대화하였습니다. 또한 텍스트 기반 리눅스 클라이언트뿐만 아니라, 사용자 편의성을 고려한 웹 클라이언트 인터페이스를 추가로 제공합니다.

#### 1.2 시스템 구성 및 개발 환경
* **서버**
    * **하드웨어:** `Raspberry Pi 4`
    * **운영체제:** `Linux` (Kernel: 6.18.29+rpt-rpi-v8)
    * **실행 형태:** 시스템 백그라운드에서 상시 동작하는 데몬 프로세스
* **리눅스 클라이언트**
    * **운영체제:** `Ubuntu 26.04` (Kernel: 7.0.0-15-generic)
    * **특징:** 프로그램 실행 중 예기치 않은 종료를 방지하기 위해 `SIGINT`를 제외한 모든 시그널을 블로킹 처리
* **웹 클라이언트**
    * **특징:** 사용자가 웹 브라우저를 통해 편리하게 장치를 제어할 수 있도록 추가 구현된 인터페이스
* **제어 대상 장치 (Target Devices)**
    * **LED:** 원격 `On/Off` 제어 및 3단계 밝기 조절(최대, 중간, 최저)
    * **부저:** 경고음 및 음악 소리 `On/Off` 제어
    * **조도 센서:** 주변 밝기 실시간 값 확인 및 조도에 따른 LED 자동 제어 연동
    * **7 세그먼트:** 타이머 숫자(`0~9`) 표시 및 카운트다운 기능, `0` 도달 시 부저 연동
* **빌드 시스템:** `Makefile`을 통한 빌드 자동화 구성

## 3. 실행 방법
프로젝트 최상위 디렉토리에서 스크립트를 통해 빌드 및 배포를 진행한 후, 클라이언트 프로그램을 실행하여 서버에 접속합니다.

⚠️ **주의 사항**: `scp.sh` 스크립트를 실행하기 전에 파일 내부의 서버 접속 정보를 본인의 라즈베리파이 환경에 맞게 반드시 수정해야 합니다.

* `scp.sh` 수정 예시:
```bash
#!/bin/bash

make
# 아래의 사용자명(mangjun), IP(172.20.33.117), 홈 디렉터리 경로를 본인 환경에 맞춰 변경하세요.
scp ./bin/librpi.so mangjun@172.20.33.117:/home/mangjun/
scp ./bin/index.html mangjun@172.20.33.117:/home/mangjun/
scp ./bin/server mangjun@172.20.33.117:/home/mangjun/
```

환경에 맞게 스크립트를 수정했다면, 아래 명령어를 순서대로 실행합니다.

```bash
# 1. 자동 빌드 및 서버 배포 스크립트 실행 (프로젝트 최상위 폴더)
./scp.sh

# 2. 홈 디렉터리에서 서버 실행 (rpi4) 
./server

# 3. 프로젝트 루트 디렉터리에서 클라이언트 실행 (Linux)
./bin/client <서버_IP> <서버_PORT>
```

### 4.2 시스템 아키텍처 및 통신 흐름
* **비동기 이벤트 루프(`core.c`)**
초기 프로토타입에서는 클라이언트가 접속할 때마다 새로운 스레드를 생성하는 멀티 스레드 방식을 사용했습니다. 하지만 제어 명령을 내리는 TCP 클라이언트와 상태를 모니터링하는 HTTP 웹 클라이언트가 동시에 다수 접속할 경우, 잦은 `Context Switching`으로 인해 라즈베리파이 환경에서 시스템 오버헤드가 급증하는 문제가 있었습니다. 이를 해결하기 위해 멀티 스레드 네트워크 수신 구조를 과감히 제거하고, 리눅스의 고성능 I/O 다중화 기법인 `epoll`을 도입하여 싱글 스레드 비동기 이벤트 루프로 서버 코어를 전면 리팩터링했습니다.
```c
while (1) {
    if ((event_count = epoll_wait(epoll_fd, events, MAX_EVENTS, -1)) == -1) {
        continue;
    }

    for (i = 0; i < event_count; i++) {
        if (events[i].data.fd == tcp_sockfd) // TCP 연결 요청 처리
        else if (events[i].data.fd == http_sockfd) // HTTP 연결 요청 처리
        else // TCP / HTTP 수신 처리
    }
}
```

* **SSE 아키텍처**
SSE는 서버에서 클라이언트로의 단방향 통신에 특화된 브라우저 네이티브 기술입니다. `Content-Type: text/event-stream` 응답 헤더만으로 기존 HTTP 인프라 위에서 손쉽게 지속 연결을 유지할 수 있어, 현재의 시스템 구조에 훨씬 가볍고 적합한 아키텍처라고 판단했습니다.

### 4.3 전역 상태 관리 및 멀티스레드 동기화
* **상태 중앙 집중화**
    * **설계 배경 및 문제 인식**
    다양한 하드웨어(LED, 부저, 세그먼트 등)를 제어하는 동적 라이브러리(.so)를 비동기 스레드로 실행할 때, 초기에는 각 모듈마다 전달해야 하는 매개변수의 형태가 달라 스레드 생성 코드의 파편화가 발생했습니다. 또한, 웹 클라이언트 환경이 추가되면서 전체 장치의 현재 상태를 한 번에 `JSON` 형태로 브로드캐스트해야 했는데, 개별 스레드나 모듈에서 상태를 각자 관리할 경우 데이터를 취합하는 과정에서 병목과 동기화 문제가 발생할 것이 자명했습니다.
    * **해결 방안**
    이를 해결하기 위해 시스템의 모든 하드웨어 상태와 제어 플래그를 캡슐화한 전역 구조체 `device_state`를 설계하여, 서버 내의 단일 진실 공급원으로 삼았습니다.
        * **인터페이스 통일 및 모듈화**: 모든 하드웨어 제어 워커 스레드가 `device_state` 포인터 하나만을 인자로 받도록 시그니처를 통일했습니다. 덕분에 dlopen으로 동적 라이브러리를 로드하고 스레드를 생성하는 중복 코드를 제거하고, 하나의 깔끔한 범용 헬퍼 함수로 추상화하여 코드의 가독성과 확장성을 극대화했습니다. 새로운 하드웨어 모듈이 추가되더라도 메인 서버 로직의 수정 없이 라이브러리만 추가하면 되는 구조를 완성했습니다.
        * **데이터 파이프라인 최적화**: 웹 서버 로직이 클라이언트들에게 상태를 전송할 때, 여러 스레드를 순회하며 데이터를 수집할 필요가 없어졌습니다. 오직 `device_state` 구조체 하나만 읽어 `JSON`으로 직렬화한 뒤 전송하면 되므로, 실시간 브로드캐스트 아키텍처가 극도로 단순해지고 응답 속도가 향상되었습니다.

* **멀티스레드 동기화 및 안정성 확보**
    * **경쟁 상태 방지**
    비동기 네트워크 이벤트 루프와 여러 하드웨어 제어 워커 스레드가 전역 구조체인 `g_state`에 동시에 접근하여 값을 읽고 쓰는 환경에서는 데이터 오염이 발생할 위험이 높습니다. 이를 차단하기 위해 상태 변경 시 `pthread_mutex`를 도입하여 모든 데이터 갱신 로직을 스레드 안전하게 보호했습니다.
    * **임계 구역 최소화 및 I/O 블로킹 방어 로직**
    단순히 락을 거는 것에 그치지 않고, 시스템 전체의 퍼포먼스 저하와 잠재적인 교착 상태를 예방하기 위해 상태 갱신과 네트워크 전송 시점을 완벽히 분리하는 플래그 패턴을 설계했습니다.

```c
// state.c: 부저 상태 업데이트 및 브로드캐스트 트리거 로직
void update_buzzer_state(const int new_state)
{
    int changed = 0;

    // 1. 임계 구역(Critical Section): 순수 메모리 갱신만 수행하여 락 점유 시간 최소화
    pthread_mutex_lock(&mutex);
    if (g_state.buzzer_on != new_state) {
        g_state.buzzer_on = new_state;
        changed = 1;    // 상태 변경 플래그 활성화
    }
    pthread_mutex_unlock(&mutex);
    // 임계 구역 종료

    // 2. 락 해제 후 안전한 상태에서 네트워크 I/O(브로드캐스트) 수행
    if (changed) {
        broadcast_sse();
    }
}
```

### 4.4 네트워크 예외 처리 및 방어적 프로그래밍
* **이중 연결 리스트를 활용한 O(1) 세션 관리**
`epoll` 기반의 싱글 스레드 서버에서는 특정 클라이언트의 연결이 종료되었을 때, 세션 리스트에서 해당 클라이언트를 빠르게 제거하고 자원을 반환해야 합니다.
    * **문제점**: 배열이나 단일 연결 리스트를 사용할 경우, 삭제할 세션을 찾기 위해 전체 리스트를 순회해야 하므로 $O(N)$의 시간 복잡도가 발생하여 서버의 이벤트 루프에 병목을 유발할 수 있습니다.
    * **해결 방안**: 클라이언트 세션 관리에 이중 연결 리스트 자료구조를 도입했습니다. 종료 요청이 감지되면, 해당 노드의 이전/다음 포인터만 재연결하여 $O(1)$의 시간 복잡도로 즉각적인 노드 삭제 및 메모리 반환(free)이 이루어지도록 최적화하여 메모리 누수와 좀비 소켓을 원천 차단했습니다.

## 5. 문제점 및 보완 사항

#### 5.1 하드웨어 이슈: 7세그먼트 모듈 핀 매핑 오류
* **문제 현상**: 1일차 단위 테스트 도중 7세그먼트 디스플레이가 정상적으로 동작하지 않고 아무런 출력이 켜지지 않는 현상 발생.

* **원인 분석**: 초기 회로 구성 시 해당 모듈을 공통 음극 방식으로 판단하여 공통 핀을 GND에 연결했으나, 실제 사용된 소자는 공통 양극 방식인 것으로 확인됨.

* **해결 방법**: 공통 핀의 연결을 GND에서 5V(VCC) 전원으로 재배치한 후, 소스 코드의 제어 신호(High/Low)를 모듈 특성에 맞게 수정하여 정상 동작을 확인함.

#### 5.2 통신 이슈: 웹 클라이언트 실시간 상태 업데이트 불가
* **문제 현상**: 웹 서버 연동 후, 서버 측에서 장치 상태 데이터를 전송하기만 해서는 웹 클라이언트 화면이 즉각적으로 갱신되지 않는 현상 발생. 사용자가 수동으로 새로고침을 해야만 최신 상태가 반영됨.

* **원인 분석**: 일반적인 HTTP 프로토콜은 클라이언트의 요청이 있어야만 서버가 응답하는 단방향 통신 구조임. 따라서 다른 클라이언트나 하드웨어 연동에 의해 장치 상태가 변경되었을 때, 서버가 주도적으로 웹 브라우저에 변경된 데이터를 밀어 넣어줄 수 없었음.

* **해결 방법**: 서버와 웹 클라이언트 간에 단방향 지속 연결을 유지할 수 있는 SSE 기술을 도입함. 이를 통해 장치 상태에 변화가 생길 때마다 서버가 연결된 모든 웹 브라우저에 JSON 형태로 이벤트를 즉시 브로드캐스트하도록 통신 구조를 개편하여, 실시간으로 UI가 동기화되도록 보완함.

> **요약**: 프로젝트의 최종 목표이자 확장 기능이었던 '실시간 웹 모니터링 및 제어 시스템'을 미들웨어 없이 순수 C 언어만으로 완벽하게 구현하며 모든 개발을 조기 완료하였습니다. HTTP 파싱과 SSE를 활용한 단방향 브로드캐스트를 구축하였고, 멀티스레드 환경에서의 Race Condition 및 데드락을 방지하기 위해 Mutex 동기화 패턴을 정밀하게 설계했습니다.

##### ✅ 완료된 작업
1. **순수 C 기반 HTTP 파싱 및 웹 서버 연동**
    * `epoll` 이벤트 루프 내부에 HTTP GET/POST 요청을 라우팅하는 로직을 직접 구현
    * 웹소켓 대신 브라우저 네이티브 기술인 SSE를 채택하여, 연결을 끊지 않고 하드웨어 상태 변화를 다수의 웹 클라이언트에게 실시간 JSON 형태로 브로드캐스트하는 구조 완성
2. **Mutex를 활용한 완벽한 스레드 동기화**
    * state.c로 전역 상태 접근을 캡슐화하고, pthread_mutex를 적용하여 여러 워커 스레드와 네트워크 루프가 동시에 상태에 접근할 때 발생하는 Race Condition 원천 차단
    * 락 안에서는 값만 갱신하고 플래그를 반환한 뒤, 락을 해제한 상태에서 브로드캐스트를 쏘는 플래그 패턴을 도입하여 데드락 위험 완벽 해소
3. **반응형 웹 대시보드 UI 구축**
    * 다크 모드 및 네온 테마를 적용한 세련된 IoT 관제 대시보드(`index.html`) 구현
    * 클라이언트 명령은 Fetch API(POST) 로 전송하고, 상태 업데이트는 EventSource(SSE) 로 수신하는 비동기 분리형 프론트엔드 통신 아키텍처 적용
